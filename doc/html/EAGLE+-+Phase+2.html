Software development : EAGLE - Phase 2

Introduction

EAGLE Phase 1 was completed in February 2012, marked by the release of EAGLE version 1.0.
Phase 2 can now start, and will expand EAGLE Simulation in several directions.

Addressing business needs

Now that we have a good base to build upon, our primary objective is to target corporate goals.
We believe that EAGLE can address a broad range of business needs and create many new opportunities.

EAGLE could address the corporate goals in the following manner:

  1. Support development's software and data analysis needs
    1. Make most other projects more efficient by providing ways to identify problems early and precisely
    2. Provide ways to identify where to put efforts in current and future Illumina projects
    3. Simulate non-existing instruments to allow early software validation and reduce future time-to-market
  2. Support competitive analysis, provide Illumina with competitive knowledge
    1. Identify where Illumina tools are better
      1. Benchmark / Compare / Find tools weak points (bwa+gatk vs Isaac?)
      2. Results can be used to publicise Illumina tools if and where they are actually better
    2. Identify and fix issues before competitors can use them against us
      1. If we don't do it first, competitors may report Illumina's deficiencies first before we could address them, then we will anyway need to spend time to verify the claims and fix our tools. Better be prepared and do it first.
  3. Quality control / FDA regulations
    1. Use data produced by EAGLE as Illumina's golden standard during validation/verification testing of the sequencers for FDA regulated projects
    2. Provide ways of quantifying tools accuracy
      1. Key feature for "FDA testing / regulatory aspect" - Understanding of what we simulate == understanding of what gets or doesn't get tested in a simulated environment
      2. for Xprize? (Advance validated definition of human genome)
  4. Make BaseSpace a successful platform (ideas only here)
    1. Provide a set of simulated datasets that expose specific properties useful for testing future BaseSpace tools
    2. Make datasets visible as if they were present, but actually generated on the fly when needed, therefore taking up no space
    3. Provide EAGLE as a "dataset generator tool" with simple user interface to feed other BaseSpace tools

Strategy

Our aim is to address the business needs listed in the previous section, and to focus efforts for best business value.

  1. Be proactive in getting more users
  2. Query those users to figure out what they need, to identify current business needs/problems
  3. Identify EAGLE developments required to solve those needs
  4. Identify which developments may help multiple other projects to advance / address multiple business needs
  5. Focus development for best business value

Criteria for success

We want to "address business needs" and "focus efforts for best business value"
So the criteria for success could be:

  • How many business needs did we address?
  • What business value did we create through our actions?

Phase 2 directions

  1. Dissemination
  2. On simulating reality, filtering reality
    1. Quantify how close to reality the simulated datasets are
      - because that's the first question everyone asks
      - because that means that'd be the most convincing argument for it
      => being able to describe precisely what is real, what is not, and why
      Why is it useful?
      - Better understanding of the realness of our simulation goes together with a better understanding of where the sequencing errors come from, which may ultimately allow us to improve the analysis tools.
      - Understanding of what we simulate == understanding of what gets or doesn't get tested in a simulated environment : Key feature for "FDA testing / regulatory aspect"
    2. Make the simulated datasets as real as possible
      Based on the previous analysis.
      Why is it useful?
      - to test features of the tools that are useful in real datasets (e.g. it may be counter-productive to optimise a variant caller to better detect deletions in a very simple simulated environment, as it may worsen the detection in a real environment).
    3. Determine which properties of a real dataset can be tested independently vs properties that have to be simulated together
    4. Simulate non-existing instruments
  3. Build an EAGLE Analysis framework around the EAGLE Simulation
    1. Specs,Design&Build Analysis framework
      - Mostly simulated items backtracking and comparison tools
      - Could also benefit from using a workflow management system
      - Will need BAM generation
    2. Use EAGLE to Benchmark / Compare / Find tools weak points (bwa+gatk vs Isaac?)
      Answers business needs:
      - useful to know where to put efforts in the future
      - useful to understand where our tools are better, and publicise our tools if they are actually better
      - if we don't do it first, others may run other simulators and report Illumina 's deficiencies first before we could address them, then we will anyway need to verify the claims, then improve our tools. so better be prepared and do it first.
    3. Simulate non-reality: identify and simulate features that lead to best analysis
    4. (Feedback from Come): Multi sample analysis
  4. Contribute to Basespace
    1. Generate a set of simulated datasets that expose specific properties useful for testing future BaseSpace tools
    2. Provide EAGLE as a "dataset generator tool" with simple user interface to feed other BaseSpace tools
  5. Features development
    Implementation of features requested by users, that don't require a grand analysis of reqs/specs cycle
    e.g. User-specified random seeds, parsing of VCF GenoType field, etc.

Focus development for best business value
